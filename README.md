# 🧠 DocMind

一个基于本地文档的智能问答系统，  
支持多格式解析（TXT/MD/DOC/DOCX/PDF/图片 OCR）、向量检索（RAG）与多轮对话。

👉 面向“分散笔记难以统一检索与理解”的问题，提供对话式整理与查询能力。

---

## ✨ 项目背景

日常使用中，笔记通常以 txt / md / doc / docx / pdf / png / jpg 等形式分散存储：

- 可以查到内容，但不方便统一查找
- 可以找到信息，但不容易整理与理解
- 可以继续思考，但不方便基于已有内容往下追问

DocMind 尝试在这些场景下提供一个更自然的使用方式。

---

## 🚀 可以做什么

### 1. 查找并简单整理内容

可以从多个笔记中找到相关内容，但不只是简单返回，还会做一些整理，例如：

- 最近在做什么
- 最近遇到哪些问题

---

### 2. 支持连续提问

可以基于上一轮结果继续提问，例如：

- “刚才提到的问题是哪个模块？”

系统会结合上下文进行回答。

---

### 3. 在已有结果上继续细化

可以在已有结果基础上继续提问，并尽量在原有范围内查找，减少偏离。

---

### 4. 对记录做简单整理

可以把分散的信息整理成更结构化的形式，例如：

- 按时间线梳理
- 汇总关键事件
- 将不同类型记录放在一起查看

---

### 5. 文件级操作（重命名 / 删除）

支持在对话中发起文件改名和删除操作，采用“预览 -> 确认 -> 执行”流程：

- 重命名：先展示变更预览，再执行
- 删除：软删除到 `.docmind_trash/日期/`，不直接物理抹除
- 变更后会同步更新当前会话里的仓库状态

---

### 6. 文件变更历史查询

支持查询近期文件变更历史（重命名 / 软删除）：

- 可按“最近 N 条”查看
- 输出包含 `event_id`、变更前后路径、SHA 摘要与时间
- 变更记录落库到本地 SQLite（`file_change_log.db`）

---
## 🧩 核心能力（工程视角）

- 基于 embedding + 向量检索 + LLM 的 RAG 问答链路
- 多轮对话状态机（支持追问 / 纠偏 / 上下文约束）
- 候选集约束策略，降低幻觉与无关扩展
- 增量索引与本地缓存机制，提升响应效率
- 本地小模型参与查询改写与标签提取（降低成本）
- 文件系统操作闭环（重命名 / 删除 / 历史追踪）

## ▶️ 运行方式

```bash
pip install -r requirements.txt
python ask_notes.py
```

可选参数与环境变量：

- `--notes-dir <path>`：指定笔记目录（优先级高于环境变量）
- `DOCMIND_NOTES_DIR=<path>`：指定默认笔记目录
- `DOCMIND_MATCH_LOG_LIMIT=<int>`：控制命中明细日志最大行数（默认 60）

👉 示例数据见：examples/demo_notes_public/

## 🧠 一些实现方式

### 对话状态的简单区分

将问题大致分为几类：

- 普通查询
- 基于内容的追问
- 整理类请求（如时间线）
- 基于结果的追问

不同类型采用不同处理方式。

---

### 查询构建的调整

根据问题类型，对检索词做一些处理：

- 初始问题尽量保持稳定
- 多轮提问保留上下文信息
- 整理类请求适当放宽限制
- 基于结果的追问尽量限制范围

---

### 避免无关扩展

在构建查询时尽量避免引入与原问题无关的词，  
同时在需要整理信息时允许一定扩展。

---

### 本地索引

- 使用本地 embedding 模型进行编码
- 支持增量更新
- 使用缓存减少重复计算

## 🛠️ 技术栈

- Python
- BGE 向量模型（本地部署）
- 基于 embedding 的语义检索
- 多格式文档加载（txt / md / doc / docx / pdf / png / jpg / jpeg / bmp / webp）
- 自实现对话状态管理（多轮问答 / 追问）
- 增量索引与缓存机制
- 文件变更记录（SQLite）与可追溯查询
